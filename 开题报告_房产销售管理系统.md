| **一、选题依据及研究意义**  **（一）选题依据**  房产销售业务链条长、参与角色多，通常涉及房源信息维护、客户意向管理、带看过程记录、交易状态流转、回款确认与佣金结算等多个环节。在传统人工或分散式工具（如Excel、纸质台账、即时通讯）管理模式下，容易出现数据冗余与不一致、审批流程缺失、状态不可追溯、权限边界模糊等问题，难以满足团队协作与经营管理的要求。尤其在房源数量与客户规模增长的情况下，人工管理方式在房源状态（在售/预订/成交/下架）维护、交易与回款关联、佣金核算与归集等方面容易产生错误，进而影响成交效率与管理决策。因此，设计并实现一套面向房产销售场景的管理系统具有现实需求与实践价值。  **（二）研究意义**  第一，提升房产销售管理效率与数据质量：系统化管理房源、客户、带看、交易、回款、佣金等核心数据，降低重复录入与人为差错，提高业务处理效率，并形成可追溯的数据链路。第二，保障业务流程规范与可控：通过交易状态机、审核与权限控制约束关键操作（如房源审核、交易创建、财务确认），降低业务风险，确保数据一致性。第三，为经营分析与管理决策提供基础：通过统计报表对成交额、成交量、回款与销售业绩等指标进行汇总展示，为销售管理与资源配置提供依据。第四，促进专业综合能力培养：课题综合运用Java Web开发、数据库设计、软件工程与前后端分离架构等知识，具有较强的工程实践意义。 |
| **二、选题研究现状**  房产销售管理系统本质上属于CRM/业务流程管理（BPM）与行业管理软件的交叉领域。国际上，成熟的销售管理与CRM系统多以SaaS形式提供，强调标准化流程、可配置权限与报表分析能力，并可通过API集成短信、邮件、电子签、支付、BI等外部服务；其优势在于可配置性强、数据分析体系完整，但往往成本较高、行业落地需要二次配置与本地化适配。国内相关研究与应用发展迅速，面向中小型门店/团队的房产中介与渠道管理系统普遍采用B/S架构，重点聚焦房源、客源、带看、成交与分佣等基础业务，技术上常使用Spring Boot、MyBatis、Vue等开源技术栈，具备低成本、开发效率高的特点。但从现有系统与研究成果看，仍存在一些共性不足：其一，流程与状态规则常以硬编码实现，可扩展性与可配置性不足；其二，数据权限与隐私保护在工程实现上容易被弱化，导致越权访问风险；其三，经营分析多停留在简单统计，缺少指标口径治理与多维分析；其四，交易、回款、佣金等跨模块数据一致性控制不够严谨，容易引发“状态不同步”的问题。基于上述现状，本课题拟在实现核心业务闭环的基础上，突出权限控制与状态机一致性设计，并提供可验证的统计报表输出。 |
| **三、拟研究的主要内容和思路**  （一）预期研究目标  1.设计并实现一套前后端分离的房产销售管理系统，覆盖房源、客户、带看、交易、回款、佣金、团队与权限等核心业务模块。  2.建立清晰的业务状态流转与一致性约束机制，实现交易状态变化与房源状态联动，并通过事务保障关键数据一致性。  3.实现基于角色的访问控制（RBAC）与数据权限过滤，满足销售顾问/经理/财务/管理员等不同角色的数据可见性与操作范围要求。  4.提供基础经营统计与可视化展示（成交量、成交额、回款、业绩排行等），支持管理决策。  （二）研究内容要点  1.需求分析：梳理房产销售业务流程与角色职责，提炼功能需求、数据需求与权限需求。  2.系统总体设计：确定系统架构、模块划分、接口规范与统一异常处理策略。  3.数据库设计：完成ER模型与表结构设计，建立房源、客户、带看、交易、回款、佣金及权限相关表之间的关系，并设计必要索引。  4.房源与客户模块实现：实现房源信息维护、筛选分页、图片管理、审核与状态管理；实现客户信息维护与归属分配。  5.带看模块实现：实现带看记录创建与查询，并关联客户与房源。  6.交易模块实现：实现交易创建、交易详情与列表，多表关联查询；实现交易状态机与关键业务规则（房源在售方可交易、同房源不可多笔进行中交易等），并实现状态联动。  7.回款与佣金模块实现：实现收付款记录登记与查询、财务确认控制；实现佣金记录生成与查询（固定比例或可配置比例），与交易关联。  8.统计报表实现：实现成交/回款/业绩等统计接口与前端展示（ECharts或表格）。  9.测试与部署：编写测试用例与测试报告，提供可复现运行说明与初始化数据脚本。  （三）采取的研究方法  1.文献调研与对比分析：查阅CRM、RBAC、状态机、行业管理系统等相关文献与案例，总结可借鉴方案。  2.软件工程方法：采用迭代式开发方式，按模块逐步实现并联调，持续测试与修复。  3.原型与场景驱动设计：围绕典型业务场景（房源审核、创建交易、回款确认、佣金生成）驱动数据库与接口设计。  （四）技术路线图  1.需求分析 → 2.总体设计（分层架构、模块划分、接口规范）→ 3.数据库设计（ER图与建表脚本）→ 4.后端实现（Spring Boot + MyBatis-Plus，事务与权限控制）→ 5.前端实现（Vue3 + Element Plus，列表/表单/统计图表）→ 6.联调与测试（功能、权限、异常、边界）→ 7.部署与文档完善（README、接口文档、初始化数据）→ 8.论文撰写与答辩准备。 |
| 研究的创新点及重、难点：  （一）创新点：1.面向房产销售场景构建“房源—客户—带看—交易—回款—佣金”闭环的数据模型与业务规则，实现交易状态与房源状态联动。2.在系统工程实现中引入RBAC与数据权限过滤（销售仅可见本人数据），提升信息安全与隐私保护能力。3.提供经营统计指标（成交/回款/业绩排行）与可视化展示，为管理决策提供基础数据支撑。（二）重点：交易状态机与跨模块一致性控制（交易、房源、回款、佣金的联动）；权限模型与数据范围控制；数据库结构与查询性能优化。（三）难点：1.复杂业务规则在代码层面的可维护实现（避免大量if-else导致难扩展）。2.多表关联查询与分页性能优化（索引设计、SQL优化）。3.并发与事务边界控制（例如同一房源交易创建的并发冲突处理）。4.前后端交互与权限错误处理的统一（统一响应、统一异常、前端友好提示）。 |
